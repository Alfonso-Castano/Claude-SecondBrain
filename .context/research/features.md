# Features Research: Personal "Second Brain" Knowledge System (Karpathy-style raw/wiki)

## Summary

The plan already covers the real table stakes for this category (ingest → wiki → index → lint, citations, trust tagging, contradiction handling) — it matches what other Karpathy-wiki implementations converged on independently. The genuine gap is a **page-level template/frontmatter convention** (summary, tags, trust tags, dates) that the lint pass needs in order to actually detect staleness and gaps — this should be locked down in Feature #1, not left implicit. The biggest risk isn't a missing feature, it's scope creep toward what other "second brain" tools in this space have shipped and Alfonso has explicitly rejected: background automation, auto-reconciled contradictions, semantic/vector search, and a visualization frontend. Every one of those shows up in a comparable public project; every one directly contradicts a stated constraint in `OVERVIEW.md`. Protect the propose-then-confirm boundary — it's the actual differentiator, not a nice-to-have.

## Findings

### Table stakes (expected — a raw/wiki system without these is broken, not minimal)

These are already in the plan; listed to confirm nothing is missing, plus two additions.

1. **Three-file-type structure: `raw/`, `wiki/`, `index.md`, `log.md`.** This is the pattern itself, confirmed by every independent implementation found (Astro-Han/karpathy-llm-wiki, AgriciDaniel/claude-obsidian, MindStudio writeups of Karpathy's original gist). — **HIGH** (cross-checked across 3+ independent sources plus Karpathy's own gist).
2. **Ingest → compile → cross-link, not just append.** New sources should update/strengthen existing pages and cross-references, not just pile on new files. Already planned via the deep-research ingestion pass. — **HIGH**.
3. **Citations on every page, tracing back to source.** Universal across implementations (`karpathy-llm-wiki`'s query op explicitly returns "citations linking back to your markdown pages"). Already planned. — **HIGH**.
4. **Lint / health-check pass**: broken links, index integrity, gaps, contradictions, stale claims. Already planned, matches `karpathy-llm-wiki`'s "Lint" operation almost verbatim. — **HIGH**.
5. **Index-first query discipline** (read `index.md`, drill in selectively) as the alternative to embeddings/RAG at this scale. Already planned. Independently corroborated: "For personal or small-team knowledge bases (up to a few hundred documents), the LLM Wiki pattern is generally simpler, more reliable, and easier to debug than RAG." — **MEDIUM** (single source, marketing-adjacent blog, but consistent with why Karpathy's pattern exists at all).
6. **Explicit conflict/open-question handling**, not silent overwrite. When lint finds a contradiction it can't confidently resolve, the convention elsewhere is to write it as its own flagged page/section rather than pick a side quietly. Already implied by "propose-then-confirm" but should be made an explicit lint behavior: *contradiction found → proposed resolution + explicit "open question" fallback if Alfonso doesn't confirm a resolution*. — **MEDIUM**.
7. **MISSING — page-level template/frontmatter convention.** Every comparable implementation standardizes on a minimal page shape: title, one-line summary, tags, last-updated date, source/citation. `OVERVIEW.md` doesn't currently specify this at the page level (it specifies folder/file structure, not per-page schema). Without it, lint has nothing structured to check for staleness or gaps against — "stale claims" detection requires *some* date field to exist somewhere on a page. **Recommend Feature #1 include a minimal page template as part of `CLAUDE-WIKI.md`'s schema**, not deferred to organic emergence. — **MEDIUM** (inferred from convergent pattern across implementations, not a single authoritative spec).
8. **MISSING (partial) — explicit backlink / "related pages" convention.** Because Obsidian-style visual graph is explicitly out of scope, cross-linking has to be discoverable through plain markdown links and, ideally, a lightweight "linked from" or "related" section on pages — otherwise cross-venture connections (a stated core value) only exist in the direction the original author happened to write the link, not bidirectionally. This is cheap to add now and expensive to retrofit across hundreds of pages later. — **LOW-MEDIUM** (reasoned from the stated cross-venture-learning goal + how other markdown-wiki tools solve discoverability without a frontend, not a directly cited source).

### Differentiators (real edge — worth explicitly protecting in the roadmap, not trading away for velocity)

1. **Propose-then-confirm as the default, with exactly one named exception.** This is genuinely rare. The comparable public projects in this exact niche (`obsidian-second-brain`, `claude-obsidian`) advertise the opposite as a selling point — "contradictions reconcile automatically," "the vault rewrites itself," "scheduled agents that maintain the vault while you sleep." Alfonso's design is a deliberate reaction against that category norm, grounded in a stated prior failure (over-delegating strategic decisions to AI). This is the single most important thing the roadmap should not let slip in the name of convenience. — **HIGH** (direct project rationale in `OVERVIEW.md`, cross-checked against the category norm found in public repos).
2. **Trust/verification tagging (extracted / inferred / ambiguous).** Borrowed explicitly from Graphify, which uses this for codebase knowledge graphs, not personal knowledge — applying it to a personal wiki is a legitimate adaptation and not something the comparable PKM tools (Mem, Reflect, Notion AI, Obsidian plugins) do at all; they treat all stored notes as equally trustworthy. Worth keeping and worth being disciplined about applying consistently (every claim, not just every page). — **HIGH** (Graphify convention independently verified via GitHub/product pages; its use in personal PKM tools absent from every comparable product found).
3. **"Grill me" active interrogation for both ingestion and reconciliation.** Comparable tools are passive — they organize what you type or paste, they don't interview you to extract tacit knowledge or challenge inconsistencies between claimed and actual outcomes. This is a real differentiator for the "executive assistant that pushes back" goal specifically, not just for capture. — **MEDIUM** (no comparable tool found doing this; conclusion is by absence of counter-examples across a reasonably thorough search, not a positive confirming source).
4. **Model of a person across ventures, not a topic-indexed archive.** Every comparable tool found (Mem, Reflect, Notion AI, Obsidian-based wikis) organizes around notes/topics/projects as the unit, not around a persistent model of the user that different projects draw on. Cross-venture transfer is structurally absent from the comparables. Keep the `About Me` file and cross-venture linking central in the roadmap — it's the actual novel value proposition, not the wiki mechanics themselves (which are now a fairly well-trodden pattern). — **MEDIUM** (inferred from the comparable products' documented feature sets, not a direct statement that this is missing industry-wide).

### Anti-features (actively avoid — found in comparable products, contradicts this project's stated constraints)

Given the design principle "no automation, no vector search, no frontend," these are not neutral omissions — they're things a later roadmap draft or a future Alfonso-in-a-hurry might be tempted to add because a comparable tool has them. Flag explicitly so the roadmap can say "considered and rejected," not silently drift.

1. **Scheduled/background agents ("maintain the vault while you sleep").** Found verbatim as a selling point in `eugeniughelbur/obsidian-second-brain`. Directly contradicts the explicit, repeated constraint: "No automation on a timer anywhere in this project... not just deferred for launch." — **HIGH** (direct conflict with an explicit written constraint; source is a real public repo advertising the feature).
2. **Fully automatic contradiction reconciliation.** `claude-obsidian`'s pitch: "contradictions reconcile automatically... the vault compounds while you sleep." This is precisely the automation-bias failure mode research warns about — when a system silently resolves conflicts, the human stops checking, and the propose-then-confirm boundary erodes by default rather than by decision. Keep every contradiction resolution behind confirmation, full stop, including ones that feel "obviously" resolvable. — **HIGH** (direct conflict with propose-then-confirm constraint; automation-bias risk independently corroborated in HITL research literature, though that literature is enterprise/general-AI-oversight context, not this project specifically — treat the general research as **MEDIUM**).
3. **Vector search / embeddings / semantic search infra.** Already explicitly out of scope. Validating that exclusion: at "up to roughly hundreds of pages" scale, index-first reading is a reasonable substitute and RAG infrastructure is widely reported as overkill and failure-prone even in production contexts, let alone a solo personal tool ("72% of enterprise RAG implementations either fail outright or deliver significantly below expectations in their first year" — enterprise context, not directly this project's scale, but directionally supportive). — **MEDIUM** (source is a single blog citing a stat without a traceable primary study; treat the number skeptically, but the directional conclusion — RAG is overkill at this scale — is corroborated by the Karpathy pattern's own stated rationale).
4. **Visualization/graph frontend (Obsidian graph view, etc.).** Already explicitly deferred, correctly. Every comparable tool's graph/visual layer is reported as decorative rather than functional once note count grows ("as long as you don't let it turn into a museum" — re: Notion AI workspaces). Nothing here suggests revisiting this deferral; the roadmap should treat "add a frontend" as gated strictly behind a concrete, felt pain point, exactly as `OVERVIEW.md` already states. — **LOW-MEDIUM** (anecdotal characterization from a comparison blog, not a rigorous study, but consistent with the project's own stated rationale for deferring it).
5. **Multi-user/collaboration features** (shared workspaces, comments, permissions) — a core Notion AI differentiator, entirely irrelevant to a single-user tool modeling one person. Don't build sharing/permissioning even in a minimal form; there is no second user.
6. **End-to-end encryption / multi-device sync infrastructure** — a core Reflect differentiator, solving a threat model (cloud storage of sensitive notes, multi-device access) this project doesn't have. `OVERVIEW.md` already settled for local git only, no remote backup — that decision is sound for this project; don't quietly add sync/encryption machinery as a "nice to have."
7. **Spaced-repetition / flashcard review flows** (seen in some PKM tools like RemNote, sometimes bolted onto "second brain" positioning). This is a memorization tool pattern; this project is a reference/reasoning tool for an AI executive-assistant, not a study aid for Alfonso himself. Out of category.
8. **Generic reusable multi-format ingestion pipeline built ahead of need** (PDF parsers, OCR, transcript pipelines as standing infrastructure). Build ingestion handling for the source types Alfonso is actually feeding in (books, articles, notes) as they show up, not a speculative universal parser framework. This is a common RAG-adjacent overbuild pattern ("chunking strategies, embedding models... introduces fragility and complexity that's often unnecessary") applied one layer up, to ingestion tooling instead of retrieval tooling.
9. **Elaborate tag taxonomy / ontology.** Keep tags to what's functionally load-bearing: trust tags (extracted/inferred/ambiguous) and the existing shared/claude-knowledge split. Resist the temptation to build a broader topic-tag hierarchy up front — comparable tools that over-invest in taxonomy tend to have the taxonomy go stale faster than the content itself.
10. **Usage analytics / dashboards.** An enterprise-RAG pattern ("building analytics feedback loops" as a core hidden cost) with no purpose in a single-user tool with a human directly in every write path. Skip entirely.

## Confidence

**MEDIUM-HIGH overall.** The table-stakes and anti-feature findings are well corroborated — multiple independent public implementations of the exact same pattern (Karpathy raw/wiki) exist and were checked against `OVERVIEW.md`'s explicit constraints, producing direct, checkable conflicts rather than inference. The differentiator claims are necessarily softer (absence-of-evidence reasoning across a search space that can't be exhaustively covered) and are marked accordingly. The one page-template gap and the backlink-convention note are the two lower-confidence, judgment-based additions — worth a light sanity-check during Feature #1 planning rather than treated as settled fact.
