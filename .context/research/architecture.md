# Architecture Research: Personal Second Brain (Karpathy raw/wiki pattern)

## Summary

The planned `raw/` → `wiki/` → `index.md`/`log.md` → `CLAUDE-WIKI.md` structure is a faithful, well-precedented implementation of Karpathy's LLM Wiki pattern (viral April 2026), which itself is being actively documented and replicated by multiple independent writeups — this is not a novel or risky architecture. The one genuine extension beyond the canonical pattern is the `shared/` vs `claude-knowledge/` paired-folder split by authority — this has no direct precedent in public LLM-wiki writeups, but it's a sound design *if* pairing is kept lazy (create the second file only when there's real independent content) rather than mechanical, because the adjacent Zettelkasten community has already hit and documented this exact duplication failure mode. Index-first retrieval is not just adequate but the *documented consensus best practice* at "hundreds of pages" scale — vector search is explicitly the wrong tool at this size, with a fairly precise, cross-sourced inflection point around 300-500 pages / when `index.md` itself gets too long to read in one pass.

## Findings

### 1. Canonical component boundaries (confirmed pattern, not this project's invention)

Multiple independent writeups (MindStudio, AAIF, natural20.com, StarMorph, a widely-cited community gist) converge on the same three/four-layer decomposition, which is exactly what this project already has:

- **`raw/`** — immutable source evidence. Never edited after creation. The agent reads it, cites it, never rewrites it. Analogous to source code in Karpathy's own "LLM is a compiler" framing (raw = source, wiki = compiled output, lint = tests, query = runtime). **HIGH confidence** — this framing is repeated near-verbatim across at least four independent sources.
- **`wiki/`** — the maintained, agent-owned knowledge layer. Cross-linked markdown pages. This is the only layer the agent is expected to actively rewrite/restructure over time.
- **`index.md`** — single catalog file, read first on every query, before touching any wiki page. Universally described as the navigation layer, not a search layer.
- **`log.md`** — append-only operation history (ingests, queries, lint passes). One source (AAIF) notes this is typically *thin* on provenance — it records that an operation happened, not a structured trace of which raw source produced which wiki claim. **This project's trust/verification tagging (extracted/inferred/ambiguous + per-page citation) is meaningfully more rigorous than the median implementation of this pattern** — worth flagging to the roadmapper as a genuine strength to protect, not a nice-to-have to cut under time pressure.
- **Schema file** (`CLAUDE-WIKI.md` here, usually `AGENTS.md`/`CLAUDE.md` in public writeups) — universally described as "the most important file in the system," encoding entity types, workflows, conventions, and quality bar. The project's decision to split this from the thin router `CLAUDE.md` is *more disciplined* than most public examples, which tend to conflate the two. **MEDIUM-HIGH confidence.**

**Build-order implication:** every source that documents this pattern treats schema-first as the correct sequencing — write the operating rules before ingesting real content, because the schema is what keeps ingestion consistent. This validates the project's own plan (Feature #1 = wiki structure + `CLAUDE-WIKI.md` content, run through full ceremony, before any real ingestion happens).

### 2. The `shared/` vs `claude-knowledge/` paired split — sound, but with a known failure mode to guard against

No public writeup of the Karpathy pattern splits the wiki layer by *authority/provenance* (human-sourced vs AI-independently-researched). The closest documented analogue is single-curator, single-agent — one person feeds sources, one agent maintains one wiki. The AAIF piece explicitly confirms the canonical pattern does not explore "federated knowledge management, source attribution, or competing authorities." So this project's split is a **genuine extension**, not something being copied from a proven reference implementation. That's not a red flag by itself, but it means the failure modes have to be reasoned about from adjacent precedent rather than cited directly.

The closest real precedent is the Zettelkasten distinction between **literature notes** (what a source says) and **permanent/evergreen notes** (your own synthesized understanding) — structurally the same "external knowledge vs. own synthesis" boundary as `shared/` vs `claude-knowledge/`. The Zettelkasten community has explicitly documented the failure mode this project should watch for:

> "Sometimes there is no need for [a second note] because the [first] note covers it well enough... do not create a duplicate note that only links to [the other]. Create [a second note] only if you encounter... additional insights." (zettelkasten.de / Zettelkasten Forum consensus)

**Concrete risk for this project:** if the paired-file convention is applied mechanically — every topic in `index.md` gets both a `shared/X.md` and a `claude-knowledge/X.md` stub regardless of whether Claude's independent research actually produced anything beyond what Alfonso said — the wiki accumulates low-information stub files that exist only to satisfy the pairing convention, diluting the index and creating false signal that a topic has been independently researched when it hasn't.

**Recommendation for the roadmapper:** Feature #1's `CLAUDE-WIKI.md` schema should state explicitly that pairing is *earned, not mandatory* — a `claude-knowledge/X.md` file gets created only when independent research actually surfaces something (a supporting source, a contradiction, an alternative framing), and until then `shared/X.md` exists alone in the index with no phantom counterpart. This is a cheap rule to write into the schema now and expensive to retrofit after dozens of stub files exist. **MEDIUM confidence** (reasoned by analogy from a well-documented adjacent field, not a source that discusses this exact pattern).

A second, related risk drawn from the same body of literature: cross-linking between paired files needs to be bidirectional and mechanically checked (lint pass), because the Zettelkasten literature separately warns that manually-maintained links between two note categories rot silently — one side gets updated, the other doesn't, and nothing surfaces the drift until a reader hits stale information. This is exactly what the project's existing lint/health-check mechanism should own; make "orphaned/unpaired cross-links between shared and claude-knowledge" an explicit lint check, not an implicit hope. **LOW-MEDIUM confidence** — inference from general PKM literature, not a source specific to AI-paired wikis.

Separately, general agent-memory literature (mem0/Atlan/Algomox-adjacent writeups) converges on a pattern that validates this project's propose-then-confirm design independent of the Karpathy pattern specifically: the recommended architecture for agent-written knowledge is "agent writes new learnings to a review/staging area, not the canonical base; a periodic human pass promotes or rejects." That is precisely the deep-ingestion-exception-aside shape this project already has. **MEDIUM confidence** (one synthesized search result drawing on several agent-memory sources, not independently cross-checked against primary docs).

### 3. Index-first retrieval vs. vector/embedding search — well-precedented at this scale, with a fairly specific inflection point

This is not a contrarian or unproven choice. It is the explicitly stated best practice at the project's scale, and the "outgrow it at hundreds of pages" framing in the project docs undersells how much headroom actually exists before switching becomes necessary:

- One implementation report (Iron Mind, an AI knowledge-base vendor writing about their own production system) states their **sweet spot is ~120-500 curated items**, with the index comfortably fitting in a single prompt at their current ~120 entries using a 200k-token context window. They explicitly reject vector search below this scale with the reasoning that "the LLM itself is the retriever. It reads English better than any embedding model." **MEDIUM confidence** (single vendor's own account of their own system, directionally consistent with other sources but not independently reproduced).
- A separate, independently-sourced estimate from a community writeup of the Karpathy pattern itself states index-based retrieval "works up to maybe 100-200 pages... beyond that, the index itself becomes too long for the LLM to read in one pass," and that the pattern is understood to **break down around 200-500 documents**, at which point a hybrid of BM25 + vector + graph traversal is the typical next step. **MEDIUM confidence** (community gist extending the original pattern, not Karpathy's own writing, but converges closely with the Iron Mind number independently).
- Cross-referencing these two independent estimates gives a fairly tight consensus band: **index-first is solid through roughly 100-300 pages, starts to strain in the 300-500 range (rising token cost per query, index file itself getting unwieldy to read), and is clearly the wrong tool north of ~500-2,000 items.** This project's own stated ceiling ("up to roughly hundreds of pages") sits comfortably inside the safe zone, not at the edge of it.

**Concrete, actionable signal for a future pitfalls/roadmap check (not needed now, but worth banking as a trigger):** don't wait for a hard page count. The trigger condition described across sources is *token cost per query rising noticeably* and *`index.md` itself no longer being comfortably readable in one pass* — both are things Claude can self-report during ordinary use ("index.md is now N lines / ~X tokens"), so a simple lint-pass check ("flag if index.md exceeds ~150-200 entries or ~3-4k tokens") is a cheap, concrete early-warning mechanism worth adding to the lint feature rather than a vague "revisit later." **MEDIUM confidence**, synthesized from two independently-sourced numeric estimates that agree closely.

### 4. Venture build-folder boundary — light-touch note only (explicitly out of scope for this build)

Per `OVERVIEW.md`, the exact connection mechanism is a separate, live research thread and this project isn't meant to resolve it. For the roadmapper's awareness only, general repo-boundary precedent (not second-brain-specific, general software engineering practice):

- **Git submodules** are the standard technical answer for "reference one repo's content from inside another while keeping histories separate," but are broadly reported (across multiple independent sources) as a poor fit for anything under active simultaneous development on both sides — detached-HEAD confusion, forgotten `submodule update`, branch-switching leaving orphaned files. **MEDIUM confidence**, this is a well-worn, widely corroborated pain point in general git usage, not specific to PKM systems.
- For a solo founder with occasional cross-references (not tight coupling), the lower-friction pattern is almost certainly **no git-level linkage at all** — separate repos, connected only by a plain-text pointer (e.g., a line in the venture's `projects/<name>/` folder noting the build repo's path/URL) that Claude can read when relevant. This avoids all submodule friction and matches the project's own stated risk rationale (keeping build artifacts, secrets, and CI/CD fully out of the wiki's git history). **LOW confidence** — this is architectural inference from general practice, not a source specific to this exact scenario; treat as a starting hypothesis for the other research thread, not a settled answer.
- Do not resolve this further inside this project's roadmap — flag it as a known dependency/open question for whichever roadmap phase first creates a real venture subfolder with an active build.

## Confidence

**Overall: MEDIUM-HIGH.** The core raw/wiki/index/log/schema structure is HIGH confidence — cross-checked across five-plus independent writeups of the same viral, actively-discussed pattern (April 2026 onward). The index-first-vs-vector inflection point is MEDIUM-HIGH — two independently-sourced numeric estimates converge closely. The shared/claude-knowledge failure-mode analysis is MEDIUM — reasoned soundly from a well-documented adjacent field (Zettelkasten) but not a source that discusses this exact AI-authority-split pattern directly, since no such precedent exists publicly yet. The venture-build-folder note is explicitly LOW confidence and intentionally light-touch, per scope.
