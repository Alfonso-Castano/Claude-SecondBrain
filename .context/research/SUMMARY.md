# Research Summary: Second Brain (AI-Agent-Operated Personal Knowledge Wiki)

## Executive Summary

This project is a faithful implementation of Andrej Karpathy's "LLM Wiki" pattern (`raw/` → `wiki/` → `index.md`/`log.md` → schema file), which went viral in April 2026 and is now documented across five-plus independent write-ups plus several working open-source implementations. The base stack — markdown files, git, Claude Code, no database, no server, no frontend — is confirmed correct and sufficient; nothing more should be added as infrastructure. Index-first retrieval (read `index.md`, drill into relevant pages, skip embeddings entirely) is not a compromise at this project's scale, it's the documented consensus best practice, with real headroom: solid through roughly 100–300 pages, only starting to strain at 300–500. All four research angles independently converge on the same build-order conclusion the project has already committed to: write the schema (`CLAUDE-WIKI.md`) before ingesting real content, because the schema is what keeps ingestion consistent — this validates Feature #1 as scoped in `OVERVIEW.md`.

Where this project genuinely departs from prior art is exactly where it's most distinctive: the `shared/`/`claude-knowledge/` authority split, trust/verification tagging, and — most importantly — propose-then-confirm governance with an AI that's expected to independently push back. None of the comparable public implementations found (SamurAIGPT/llm-wiki-agent, Ar9av/obsidian-wiki, AgriciDaniel/claude-obsidian, eugeniughelbur/obsidian-second-brain) implement any confirm/approval layer at all — several actively advertise the opposite ("the vault rewrites itself while you sleep," "contradictions reconcile automatically") as a selling point. That means the wiki mechanics (ingest/lint/index) have terrain-tested prior art to lean on, but the propose-then-confirm + pushback design does not — it has to be invented, and peer-reviewed research on automation bias suggests it can fail in a counter-intuitive direction: well-sourced, independently-researched AI pushback tends to increase human deference, not provoke harder thinking (one radiology study found expert accuracy dropped from 82% to 45.5% specifically when clinicians deferred to a wrong AI suggestion). This is the single highest-value place for design scrutiny in the whole project.

The recommended approach is not a new phase structure beyond what `OVERVIEW.md` already lays out, but a widened Feature #1: alongside the already-planned folder structure and workflows, close two concrete, currently-real gaps before ingestion starts — a page-level frontmatter/template convention (needed for lint to have anything structured to check staleness against) and an explicit lint checklist (orphan pages, citation-chain provenance, index/page drift, index size triggers, stub-file bloat from mechanical pairing). Both are cheap to write into `CLAUDE-WIKI.md` now and expensive to retrofit once dozens of pages exist. The reconciliation/pushback mechanism deserves the deepest first-principles design attention during Feature #1's discuss phase specifically because it's the one part of this project with no proven mitigation pattern to borrow.

## Key Findings

**Stack (confirmed prior art + concrete additions):**
- Markdown + git + Claude Code is the correct, sufficient stack — no DB, server, or build pipeline needed. **HIGH**
- Use standard relative markdown links (`[Topic](../claude-knowledge/topic.md)`), not `[[wikilink]]` syntax — wikilinks only resolve in tools with a resolver (Obsidian, Foam, Dendron), none of which are in scope. Claude Code will plausibly default to wikilink syntax from PKM training data unless explicitly told not to — needs a stated rule in `CLAUDE-WIKI.md`. **MEDIUM-HIGH**
- Adopt YAML frontmatter now — it's the natural, tool-agnostic home for the trust tags and citations already decided, and gives lint a deterministic, cheap thing to check before spending AI judgment on semantics. **HIGH**
- Add `markdownlint-cli2` + a link checker (`markdown-link-check` or `lychee`) as the first, fast, non-AI step *inside* the manually-triggered lint pass — not a git pre-commit hook (that would reintroduce schedule-adjacent automation). This is consistent with the no-automation-on-a-timer constraint since it's invoked by a human action. **HIGH**
- Structured git commit messages (`ingest: <source> → <pages>`, `lint: fix N stale claims`, `reconcile: ...`) give a free, complementary provenance layer alongside `log.md`. **LOW-MEDIUM**
- PARA doesn't map onto this project's structure and shouldn't be imported; Zettelkasten's atomic-note discipline is worth stating as an explicit page-granularity norm for the new-page-vs-edit rule.

**Features (table stakes confirmed, one real gap, differentiators to protect, anti-features to explicitly reject):**
- Table stakes (ingest→compile→crosslink, citations, lint, index-first, explicit conflict handling) are already planned and match every independent implementation found. **HIGH**
- **Real gap:** no page-level template/frontmatter is currently specified at the per-page level in `OVERVIEW.md` — this should be locked into Feature #1's schema, not left to emerge organically, since staleness/gap detection needs a date field to check against. **MEDIUM**
- Lower-confidence gap: an explicit backlink/"related pages" convention, since there's no Obsidian graph to make cross-links discoverable bidirectionally — cheap to add now, expensive to retrofit at scale.
- Real differentiators to protect in the roadmap: propose-then-confirm as default (comparable tools do the opposite), trust/verification tagging (absent from all comparable PKM tools — Mem, Reflect, Notion AI), "grill me" active interrogation (comparable tools are passive), and a persistent cross-venture model of Alfonso rather than a topic-indexed archive (every comparable tool organizes around notes/projects, not a person).
- Anti-features found in comparable products that directly conflict with this project's stated constraints and should be explicitly rejected, not silently avoided: scheduled/background agents, fully automatic contradiction reconciliation, vector search/embeddings, a visualization frontend, multi-user/collaboration features, e2e encryption/sync infrastructure, spaced-repetition flows, a speculative universal ingestion pipeline, an elaborate tag taxonomy, and usage analytics/dashboards.

**Architecture (canonical pattern confirmed, one genuine extension flagged with a known failure mode):**
- The `raw`/`wiki`/`index`/`log`/schema component boundaries match the canonical pattern closely; this project's trust-tagging discipline and the thin-router/detailed-schema split are *more rigorous* than the median public implementation — a real strength worth protecting under time pressure, not a nice-to-have. **HIGH**
- The `shared/` vs `claude-knowledge/` authority split has no direct public precedent — it's a genuine extension. The closest analogue (Zettelkasten literature-notes vs. permanent-notes) has a well-documented failure mode directly applicable here: mechanical pairing produces low-information stub files that dilute the index and fake the appearance of independent research. **Recommendation: make pairing earned, not mandatory — write this into `CLAUDE-WIKI.md`'s schema now.** **MEDIUM**
- Cross-links between paired files can rot silently (one side updates, the other doesn't); make "orphaned/unpaired cross-links between shared and claude-knowledge" an explicit lint check. **LOW-MEDIUM**
- General agent-memory literature independently validates the propose-then-confirm shape: staging-area writes plus periodic human promote/reject is the recommended architecture for agent-written knowledge generally, not just this project's own reasoning. **MEDIUM**
- Index-first retrieval has a fairly tight, cross-sourced consensus band: solid through ~100–300 pages, strains at 300–500, clearly wrong past ~500–2,000. **Actionable, concrete trigger for the lint feature:** flag if `index.md` exceeds ~150–200 entries or ~3–4k tokens, rather than waiting for a vague future "revisit." **MEDIUM-HIGH**
- Venture build-folder connection mechanism is out of scope for this build (per `OVERVIEW.md`), but light-touch signal from general git practice: a plain-text pointer is likely lower-friction than a git submodule for a solo founder's occasional cross-references. Flag as a known external dependency for whichever future phase creates a real venture subfolder — don't resolve it here. **LOW**

**Pitfalls (decay risks have precedent and known mitigations; propose-then-confirm/pushback risks do not):**
- Decay risks are well-precedented and solvable with discipline already anticipated by the design: cross-reference drift on ingest (index update must be atomic with the page write, never deferred), the deep-ingestion exception being the single highest-risk write path (apply trust tags immediately, treat these pages as first-priority lint/verification candidates), negative-value derivative pages that grow larger than their source (default new-page-vs-edit toward merging), stale claims with no timer to force a revisit (anchor lint triggers to natural workflow checkpoints — end of ingestion, start of new venture work, reconciliation firing — and make time-since-last-lint visible), trust tags going unrespected downstream (query workflow must surface the trust tag whenever a claim is cited back in conversation, not just when the page is opened), and index-first discipline degrading as the index grows unstructured. **MEDIUM-HIGH**
- Risks specific to this project's own novel design, with **no comparable implementation to study**:
  - Automation bias: independently-researched AI pushback plausibly *increases* deference rather than provoking harder thinking (peer-reviewed evidence, expert accuracy dropped 82%→45.5% deferring to wrong AI advice). **Primary mitigation (per Alfonso's review of this finding, now recorded in `.context/DECISIONS.md`):** present pushback as multiple competing perspectives that actively debate each other — e.g. Claude playing devil's advocate against its own synthesis, or multiple sub-agent stances — rather than a single confident, well-researched verdict to accept or reject. This is a concrete technique already described in the reference material's own guidance for using Claude as a thought partner, not an invented mitigation. Secondary/complementary mitigation: require Alfonso to articulate his own counter-reasoning during confirm, not just accept/reject; extend the "grill me" interrogation posture to pushback moments specifically, not just capture. **HIGH** on the general phenomenon, **LOW-MEDIUM** on this project's specific manifestation.
  - Self-referential/unfalsifiable "independent research": LLM citation fabrication rates run 18–69% depending on model/query type; a wiki claim from a shaky web search can get treated as settled fact by a later pass, creating circular evidence. Mitigation: lint must verify every `claude-knowledge` page's citation chain bottoms out *outside* the wiki (raw file, real external URL, or dated conversation) — not in another wiki page. **HIGH**
  - Confirmation fatigue / rubber-stamping — empirically distinct from automation bias, driven by volume and repetition rather than trust. Mitigation: differentiate confirm friction by stakes (lightweight diff for ad-hoc filing vs. fuller context for a trust-tag change or contradiction) — resolve explicitly during Feature #1's discuss phase. **MEDIUM**
  - The explainability paradox (reading Claude's reasoning substitutes for actually verifying it) and accountability diffusion in a solo system (no second reviewer to catch an error, so mistakes become a shrug) are both real but have no clean technical fix — mitigations are process-level: force a "what would make this wrong" line on every proposal, and have `log.md` capture a one-line reason/confidence note at confirm time, not just that a write happened. **MEDIUM**
  - Self-inflicted batch fatigue (Alfonso batching many reviews into one sitting, same mechanism as content-moderator fatigue, just self-scheduled) — worth a soft batch-size convention (flag sessions queuing 10+ proposed writes). **MEDIUM**

## Roadmap Implications

This project doesn't need a sprawling multi-phase roadmap — `OVERVIEW.md` already correctly scopes it around one heavily-deliberated feature (the wiki structure itself) followed by ordinary use of the existing feature loop. Research across all four dimensions reinforces rather than complicates that shape: schema-first sequencing is universal in every prior-art source, and there's no dependency chain here that requires inventing new phases.

**Recommendation: two phases, not more.**

1. **Feature #1 — wiki structure + `CLAUDE-WIKI.md` schema (full discuss/plan/execute/verify ceremony, as already mandated).** Widen its scope slightly to absorb the concrete gaps this research surfaced, rather than treating them as later add-ons:
   - Page-level YAML frontmatter template (title, folder, tags as a real list, created, last_updated, source, confidence).
   - Explicit standard-relative-markdown-links rule (anti-wikilink).
   - New-page-vs-edit rule defaulting toward merge/edit.
   - `shared`/`claude-knowledge` pairing declared *earned, not mandatory*.
   - A concrete lint checklist: markdownlint-cli2 + link-checker as pre-flight; orphan-page detection; index-entry-vs-page-content drift; citation-chain-bottoms-out-external check; index.md size/token trigger (~150–200 entries or ~3–4k tokens); trust-tag enforcement in the query workflow.
   - Propose-then-confirm workflow differentiated by stakes, with a required "what would make this wrong" line on proposals.
   - `log.md` convention capturing a confirm-time reason/confidence note, not just that a write happened.
   - The reconciliation/pushback mechanism designed around multiple competing, debating perspectives (devil's-advocate framing, not a single authoritative verdict) as the primary defense against automation bias — see `.context/DECISIONS.md`, "Pushback mechanism must present multiple debating perspectives" — with Alfonso articulating his own counter-reasoning during confirm as a secondary layer, extending "grill me" posture to pushback. This sub-piece deserves the most first-principles design time in the whole feature, since no prior art exists to validate a confirm/pushback layer against, even though the multi-perspective technique itself is source-grounded.
   - A soft batch-size convention for confirmations.

2. **First real deep-ingestion pass** (re-ingesting *$100M Offers* and the negotiation book properly through the finished workflow) — naturally sequenced after Feature #1 since the schema must exist first, and warrants full ceremony given the research judgment involved, exactly as `OVERVIEW.md` anticipates.

Everything after that — routine lint/health-check passes, ad-hoc conversational filing, reconciliation loops, future ingestions — is steady-state use of the already-existing `/feature-quick` and feature-loop tooling, not roadmap territory. If a roadmap document is produced, it should stay lightweight (these two phases) rather than inventing additional structure to fill a template. The venture build-folder connection mechanism should be noted as an external, already-flagged open dependency — not something this roadmap resolves or blocks on.

## Confidence

**Overall: MEDIUM-HIGH**, with one clearly named gap.

- The core architectural pattern (raw/wiki/index/log/schema, index-first retrieval, schema-first build order) is **HIGH** confidence — cross-verified across five-plus independent write-ups and several working implementations of the same viral, actively-discussed pattern.
- Generic tooling recommendations (YAML frontmatter, markdownlint-cli2, link checkers, standard markdown links) are **HIGH** confidence as broadly standard practice, **MEDIUM** in their exact fit to this project, since no source discusses this precise combination applied to a personal AI-authority-split wiki.
- The `shared`/`claude-knowledge` failure mode and the index-size lint trigger are **MEDIUM** — reasoned soundly by analogy (Zettelkasten, cross-sourced numeric estimates) but not directly precedented, since no public implementation splits a wiki by authority the way this one does.
- **The named gap:** the propose-then-confirm and pushback design — this project's most distinctive and important feature — is backed by **HIGH**-confidence general research (peer-reviewed automation bias and citation-fabrication findings) but only **LOW-MEDIUM** confidence on how those risks specifically manifest in *this* design, because literally no comparable public implementation has a confirm layer or an independent-pushback mechanism to study. Treat Feature #1's reconciliation/pushback sub-design as first-principles work requiring real scrutiny during the discuss phase, not as adoption of an established, de-risked pattern.
