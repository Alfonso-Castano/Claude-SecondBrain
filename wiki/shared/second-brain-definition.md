## The core idea

A second brain is a persistent, external store of knowledge that an AI agent can read from and write to, so that talking to the AI compounds over time instead of resetting every conversation.

The problem it solves: normal AI chats are **ephemeral**. Whatever gets built up in a conversation — context, decisions, nuance — disappears the moment the conversation ends. Without a system to capture it, every new conversation starts from zero, and that forces constant re-explanation of the same background, decisions, and context, over and over. A second brain turns that repeated, re-explained knowledge into something that compounds like interest in a bank, instead of evaporating every time a chat window closes.

At the most basic level, a second brain is just **markdown files organized in folders** — nothing more exotic is required to get real value. No database, no vector store, no complex infrastructure is a prerequisite. The mental shift this requires is treating the AI less like a series of disconnected tool sessions, and more like a colleague who accumulates real, permanent knowledge of you and your business over time — closer to a co-founder than a stranger you have to re-brief on every call.

## The actual job of a second brain

A second brain is a place to save notes, meeting recordings, message threads, and other raw information, structured so that both the person *and* their agent can reliably find it again later.

The test for whether it's working isn't "did I save it" — it's **"can I find it again, and can my agent find it again."** If the answer is no, the folder architecture and routing aren't set up correctly, no matter how much has been saved. A second brain that only satisfies the first half of that test (things get saved) but not the second (things get found) has failed at its actual job, even if it looks complete from the outside.

## The key mindset: work backwards from retrieval

Before deciding how to store something, the source material's advice is to think first about how it will be *queried* later — because how something will be retrieved determines how it should be stored in the first place, not the other way around.

The analogy used directly in the source material: you wouldn't design a square ball because it needs to fit through a round hoop. The shape of retrieval determines the shape of storage — not personal taste about how information "should" be organized, and not whatever structure happens to be easiest to write to at capture-time. This is a design-time question, not an afterthought: the first thing to decide when adding a new kind of information isn't "where does this go" but "how will I or my agent ever ask for this again."
